# 记忆沉降：一种自进化的四层记忆 Agent 架构

借鉴人类的记忆与行为系统，我构想了一种四层记忆架构的 Harness Agent。下文先阐述这四层架构分别是什么，再讨论它如何支撑 agent 的自我进化。

大模型近年来在上下文窗口上取得了巨大突破，从最初的几千 token 到如今百万 token 级别的 context 已成为现实。但 context 不可能无限扩展——不仅是 API 成本和推理延迟的现实约束，长上下文中的注意力稀释、检索精度下降、以及"大海捞针"式的信息定位困难同样是硬瓶颈。百万上下文已经足够覆盖绝大多数任务所需的瞬时信息量，继续卷 context 长度边际收益急剧递减。**接下来的竞争高地不是更大的 context，而是更聪明的记忆系统**——如何判断什么信息值得保留、以什么形态保留、在什么时机注入有限的 context。这正是四层记忆架构要解决的问题。


### Agent 的四层记忆架构

AI Agent 应当具备四层记忆，由近及远、由精确到模糊、由灵活到固化：

1. **工作记忆**（瞬时记忆）
2. **短期精确记忆**
3. **长期模糊记忆**
4. **超长期肌肉记忆**

**工作记忆**是四层中容量最小、成本最高的，但它唯一直接与大脑（LLM）交互——在 Agent 中，就是当前对话的 **context window**。工作记忆不做持久化，对话结束即清空，正因如此，它的每一字节都应该花在刀刃上。所有持久记忆存在的意义，归根结底都是**在正确的时刻、以最小的代价，把正确的信息加载到工作记忆**。

**短期精确记忆**容量次小，但最为精确，可变性也最强。它存储的是"如何做一件事"的完整指令——精确、可执行、但也因此每次调用都消耗可观的 context token。它适合频繁迭代、尚未固化的知识。

**长期模糊记忆**容量大幅提升，代价是细节逐渐模糊。它不再提供逐字逐句的精确指令，而是存储经过压缩的经验片段和通用原则。调用时只注入检索命中的少量片段，context 占用远低于 Skill。它的价值不在于精确复现，而在于**泛化匹配**——用模糊换来对多变场景的适应力。

**超长期肌肉记忆**则是完全固化的——调用时完全不占用工作记忆。它存储的不再是"知识"或"指令"，而是**能力**——一段预先编译好的、可重复执行的程序。大脑不需要知道它内部如何运作，只需知道何时调用、传什么参数。

四层之间的关系如下：

```
工作记忆（context window）  ← 大脑（LLM）在此思考
    ↑ 按需加载
短期精确记忆（Skill）        ← 大脑主动调取精确指令
    ↑ 降级/归档
长期模糊记忆（RAG）          ← 检索后注入工作记忆
    ↑ 固化
超长期肌肉记忆（Tool）       ← 调用但不占工作记忆
```

这张图揭示了一个清晰的梯度：从上到下，存储容量递增，调用成本递减，可变性递减，而固化程度递增。Agent 的进化，本质上就是让信息和能力沿着这个梯度**向下沉降**。

映射到当下主流的 Agent 设计（Claude Code、OpenClaw 等）：

- **工作记忆**即 context window。LLM 在此汇聚 Skill 全文、RAG 检索结果、Tool 调用结果、对话历史——所有信息最终都在这里汇合，LLM 在这里完成理解和推理。但它极其昂贵且挥发性强，对话结束即消失。
- **短期精确记忆**即 Skill。Skill 由精确的文本指令构成，随每次对话迭代可能被修改，灵活性极高——但也因此每次都消耗可观的 context token。一个复杂的 Skill 动辄数千字，全部加载进工作记忆是不可持续的。
- **长期模糊记忆**即 RAG 数据库。检索方式是模糊匹配，但容量远超 Skill 文档。值得反复强调的是，"模糊"在这里并非缺陷：模糊意味着**泛化能力**。一条 RAG 条目能覆盖多种相关场景，而 Skill 换一个变量名就可能匹配不上。这是长期记忆区别于短期记忆的核心价值——**用精度换广度**。
- **超长期肌肉记忆**即 Tool 系统（含 MCP 协议挂载的外部工具）。工具行为固定，执行过程不消耗 LLM token，调用方只需知道接口签名和功能描述。一个完善的 Tool 生态，本质上就是 Agent 的"肌肉记忆库"——越是高频使用的功能，越应该以 Tool 形态存在。

那么大模型本身呢？它不属于四层中的任何一层。大模型没有持久记忆——训练完成后权重就冻结了，不会因为一次对话而改变。它只是**大脑**——一个纯粹的信息处理器。记忆系统是独立于大脑的外部结构，正如人类的记忆并不存储在神经元的胞体里，而是分布在突触连接和外部认知工具中。


### 检索路由：大脑如何找到对的记忆？

四层记忆就位后，Agent 面对一个任务时，必须先回答一个元问题：**去哪个记忆层找信息？**

这不是一个可以忽视的细节。记忆层数越多，检索路径越多，如果缺乏路由机制，检索开销本身就会吞噬记忆系统带来的收益。想象一下：Agent 每接到一个任务都要依次检索所有 Skill、扫一遍 RAG、再检查所有 Tool 的签名——这些操作本身就消耗了大量 context 和推理时间。

可选的路由策略有三：

- **自顶向下**：先查 Skill（精确命中），未命中则查 RAG（模糊匹配），仍未命中则由 LLM 自行推理。工作记忆和 Tool 不需要"检索"——前者是当前正在处理的上下文，后者通过函数名直接调用。这个策略最简单，但存在顺序依赖：如果路由在 Skill 层误判了"已命中"，就不会继续检索 RAG，可能错过更泛化的经验。
- **并行检索**：Skill 和 RAG 同时检索，合并候选结果后由 LLM 裁决采纳哪些。避免了顺序依赖，但增加了单次检索的计算开销和 context 占用。适合对延迟不敏感、对召回率要求高的场景。
- **索引层**：维护所有 Skill 和 Tool 的轻量摘要索引（每条一两句话），先查索引定位，再按图索骥。这是最可扩展的方案——索引本身是一个极小的、常驻工作记忆的路由表，随着 Skill 和 Tool 数量增长，索引层的价值越明显。

检索路由是四层架构能否真正工作的关键基础设施。没有它，记忆不是资产，是负担。


### 四层记忆与 Agent 的自我进化

引入四层记忆架构的核心动机，是让它支撑 agent 的**自我进化**。进化的方向是：经验和能力从上层向下层逐步沉淀，每沉一层，context 成本就降一档。

这很像人类掌握一门技能的过程：一开始你需要对照详细的教程（Skill），每一步都不能错；熟练之后，你只需要记住几个关键要点和常见陷阱（RAG）；最终你形成了肌肉记忆，做事时根本不用想步骤，手自己会动（Tool）。Agent 的进化路径与此同构：

1. **Skill 化**：Agent 执行任务，积累经验，这些经验最初以精确文本的形式沉淀在短期精确记忆中（如 skill.md）。这个阶段的特征是**高保真、高成本**——每一条经验都是完整的自然语言指令，加载即占 context。
2. **RAG 化**：经验逐渐固化、不再频繁变动后，skill 文档被迁移到 RAG 数据库，转为长期模糊记忆。迁移时不是原封不动地搬运，而是经过压缩提炼，去掉一次性细节，保留可复用的原则。相比 Skill 的全文加载，RAG 仅注入检索命中的片段，context 占用大幅降低。这个阶段的特征是**中保真、中成本**。
3. **Tool 化**：当某项能力的调用频率足够高、模式足够稳定、输入输出契约足够清晰时，Agent 可将其用代码固化为 Tool，成为超长期肌肉记忆。此后调用只需知晓功能名和参数，内部实现完全移出 context——LLM 不再需要"理解"它如何工作，只管调用。这个阶段的特征是**低保真（对外部）、零成本**。

这三步形成一个完整的进化周期。随着 Agent 被持续使用，它的长期记忆和肌肉记忆不断增厚，短期记忆的容量则保持在一个稳定范围。Agent 能做的事情越来越多，但思考负担并不等比增长——这才是进化。


### 进化流程的细化

#### 迁移由谁触发？

上文描述了进化的"效果"，但未回答"时机"和"决策者"。三种可能的机制：

- **Agent 自主判断**：每次任务结束后，Agent 自行评估"这段经验是否可以归档了"，自行写入 RAG；当某类 RAG 条目被反复命中时，主动提示用户是否生成 Tool。这是最贴近"自我进化"理念的模式，但判断难度也最高——Agent 凭什么判断什么值得归档？归档的粒度是一整个 Skill 还是其中的某个步骤？判断错了怎么办？
- **Harness 层自动处理**：框架监控 Skill 的调用频率和内容稳定度。当某个 Skill 连续 N 次执行未被修改时，自动触发归档流程。规则明确、可预期，但 Harness 只能判断"是否稳定"，无法判断"是否值得记住"——一个稳定但无用的 Skill 也可能被归档。
- **用户手动触发**：用户通过 `/archive` 或 `/solidify` 等命令手动推进。最可控，每次迁移都经过人类判断，但依赖人的自觉和主动性，Agent 无法独自成长。

三种机制可以共存，但更值得关注的是触发机制的**自我进化**：前期 Agent 能力有限时，应由用户手动触发归档，确保每次迁移都经过人类判断；随着 Agent 积累的经验增多、判断力增强，再逐步将触发权下放给 Harness 自动处理，最终过渡到 Agent 自主判断。触发机制本身，也在四层记忆的进化管道中逐级下沉。

#### Skill → RAG：中间需要一步压缩

并非所有 Skill 都适合原封不动地塞进 RAG。有些 Skill 包含一次性参数、特定环境变量或某个项目的专属路径，直接归档等于制造噪音——下次检索到这段内容时，Agent 可能被无关细节误导。

迁移时，Agent 应做一次**压缩/总结**：将多次执行同类的经验提炼为一条通用原则。好的压缩保留三样东西：**关键决策点**（在哪个岔路口走哪边）、**常见失败模式**（什么情况下会出问题、怎么避免）、**适用条件**（什么场景下这条经验有效）。同时剥离具体的一次性细节（如"上次部署时用的分支名是 feature/xxx"）。

举例来说，一个部署 Skill 可能包含几十行具体步骤，压缩后的 RAG 条目可能只有三句话："该服务部署需确保 Node 版本 ≥ 20；构建产物在 /dist 目录；环境变量通过平台 Dashboard 而非 .env 文件注入。"信息密度大幅提升，适用范围也更广。

#### RAG → Tool：代码生成是最大难点

从一段模糊检索的文本自动生成一个可运行的 Tool，意味着 Agent 需要跨越一个巨大的鸿沟——从"知道怎么做"到"把它变成代码"。具体而言需要以下能力：

- **接口抽象**：从 RAG 条目中的自然语言描述，准确推断该功能的输入输出契约——参数名、类型、是否必填、默认值。接口设计一旦出错，Tool 就无法被正确调用。
- **代码生成**：生成正确且可维护的代码，而非一次性的脚本。Tool 代码需要处理正常路径和异常路径，不能假设输入总是合法的。
- **边界与异常处理**：思考"什么情况下会失败"——超时、权限不足、网络中断——并在代码中优雅地处理这些情况，返回结构化的错误信息而非裸奔溃。
- **沙箱验证**：在受控环境中运行生成的 Tool，用真实或模拟的输入验证输出是否符合预期。未经验证的 Tool 直接上线，等同于把未经测试的代码部署到生产环境。
- **版本与兼容性**：Tool 上线后可能被多次调用，其行为必须稳定、可预期。如果后续更新了 Tool 的实现，需要确保旧调用方式不突然断裂。

这本质上是 **Agent 的自我编程**——四层架构中最难自动化的一环。近期的可行路径是"半自动"：Agent 生成 Tool 代码草稿和测试用例，用户在沙箱中验证后确认上线。

但如果这一步真的做到了——Agent 不仅能调用工具，还能**为自己创造工具**——那在我眼中就是真正的智能。纵观人类文明史，从石器到蒸汽机到编译器，每一次质的飞跃都不是因为人类变得更聪明了，而是因为我们发明了新的工具来延伸自身的能力边界。Agent 从"使用工具"到"发明工具"，是同样量级的质变。一个能为自己制造工具的 Agent，不再是工具本身，而是一个**工具制造者**。

#### 进化中的版本控制

Skill 归档到 RAG 后，原 Skill 是删还是留？RAG 检索出过时信息怎么办？当 Tool 依赖的外部 API 发生变更时，Tool 如何感知并更新？

这些都不是一次性问题，而是长期运行中持续存在的张力。建议每一层的内容都携带时间戳和来源追溯：

- RAG 条目记录"来自哪个 Skill、何时归档、归档时的版本号"
- Tool 记录"来自哪些 RAG 条目、何时固化、依赖的外部 API 版本"
- 当底层信息被标记为过期时，上层固化产物能追溯根源、随之更新或标记为废弃

更进一步，这套追溯链可以支持**回退**：如果一个 Tool 被发现有问题，可以沿追溯链定位到是哪个 RAG 条目的信息有误，然后降级回 RAG 模式甚至 Skill 模式继续工作，同时修复 Tool。记忆的进化不是单向的"写入→遗忘"，而是一个可追溯、可回滚、可修复的链。

#### 实例：部署到 Cloudflare Pages 的进化全程

以下用一个具体场景完整演示三层进化：

1. **Skill 阶段**：用户编写 `deploy-cloudflare.md`，详尽描述部署步骤——"先确认本地 Git 状态干净，再确认当前分支是 main，运行 `npm run build`，构建产物在 /dist 目录，用 Cloudflare Dashboard 设置环境变量，最后触发部署……"。每次部署时 Agent 全文加载这几十字指令，逐条对照执行。这个阶段的好处是精确，代价是每次部署都消耗完整 context。

2. **RAG 阶段**：执行约 10 次后，Agent 发现每次执行的核心模式是相同的，区别只在于项目名和环境变量值。它将稳定经验压缩归档——"Cloudflare Pages 部署要点：Git 分支必须为 main；build output 目录为 /dist；环境变量通过 Dashboard 设置而非 .env 文件；部署触发方式为 Git push 或 Dashboard 手动触发"。原 Skill 精简为一句话的指针："部署时参考 RAG 中的 Cloudflare 部署经验"。每次部署的 context 开销从几百字压缩到几十字，Agent 仍然能正确部署，因为它从 RAG 中拿回了关键要点。

3. **Tool 阶段**：再执行约 50 次、确认部署模式稳定不变后，Agent 生成 MCP Tool `deploy-to-cloudflare`，接口如下：接收 `--project-path`（项目路径）、`--env-vars`（环境变量列表）、`--branch`（目标分支，默认 main），内部封装所有部署逻辑——Git 状态检查、构建、环境变量注入、部署触发、结果轮询。此后部署时 Agent 只需调用 `deploy-to-cloudflare --project-path . --env-vars "KEY1=val1,KEY2=val2"`，**零 token 理解部署流程**。LLM 甚至不需要知道部署是怎么完成的，它只需要知道"有一个 tool 能做这件事，参数是什么"。

这条进化路径能否走通，取决于三个判断标准：**执行次数阈值**（多少次才算"稳定"——太少则 Tool 质量不足，太多则进化太慢）、**正确性验证**（生成的 Tool 如何证明自己是对的——需要自动化的测试和人工的 spot check）、以及**回退机制**（Tool 出问题时能否降级回 RAG 甚至 Skill 继续工作，而非直接宕掉）。


### 结论

四层记忆系统——工作记忆、短期精确记忆、长期模糊记忆、超长期肌肉记忆——各自承担不同的记忆形态：从昂贵的精确指令，到压缩的泛化经验，到零成本的固化能力。四层之间由上至下形成一条渐进固化的进化管道，Agent 在使用中不断将信息向下沉降，释放出有限的工作记忆空间，从而支撑越来越复杂的能力。

当这套记忆系统与 Agent 的自我编程能力结合——Agent 不仅能记住经验，还能将经验固化为自己创造的工具——我们就接近了 Agent 从"被使用者"到"自主进化者"的转折点。这应当是下一代 AI Agent 架构的核心方向。
